**Findings**

- **Critical: matchbox outages/timeouts can be counted as clean validation.** In [scripts/e2e_harness.py](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:166), `validate()` ignores `code`. If `post()` returns `None, {"_error": ...}` from a timeout/down server, `errs=[]`, so line 178 increments `clean`. Fix: if `$validate` status is not 200 or body is not an OperationOutcome, record `validation_env_fail` and fail/abort the run.

- **Critical: missing `meta.profile` is misclassified as environmental.** Lines [160-165](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:160) count no profile as `env`. That hides a mapper conformance bug, especially for FML resources. Fix: make `missing_profile` its own content/structural failure and gate the verdict on it.

- **High: `ENV_MARKERS` is too broad and can hide real conformance failures.** Lines [43-55](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:43) include phrases like `unable to validate code`, `the code cannot be validated`, and `could not validate profile`; those can mean “invalid code/wrong profile,” not just tx-server noise. Fix: classify as `env`, `content`, or `ambiguous`; only auto-env known terminology-unavailable patterns, and run a tx-backed validator before publishing PASS claims.

- **High: env-only diagnostics are not actually retained.** The comment says raw samples are kept at [39-40](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:39), and the report claims they are in `results.json` at [325-327](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:325). But samples are only appended for content errors at [169-174](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:169). Fix: store every OperationOutcome issue per resource with full diagnostics, expression, severity, classifier, and classifier reason.

- **High: validation trusts the mapper’s first declared profile.** Lines [160](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:160) and [166](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:166) validate only `meta.profile[0]`. A wrong/easier first profile can pass, and additional profiles are ignored. Fix: assert expected MII profile(s) from a resource-type/path matrix, validate all declared profiles, and fail if the expected profile is absent.

- **High: no Bundle-level structural validation unless ingest is enabled.** `ingest()` checks transaction shape at [139-146](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:139), but `validate()` only checks resources. With `--no-ingest`, an empty or non-transaction Bundle can look acceptable. Fix: always run a local structural check: `resourceType=Bundle`, `type=transaction`, non-empty `entry`, unique `fullUrl`, every `entry.resource`, every `entry.request.method/url`, and resolvable intra-bundle references.

- **High: HAPI ingest success is too weak.** At [147-151](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:147), any dict with HTTP 200/201 is `ok`, even if it is not a transaction-response Bundle. It also labels both 200 and 201 as `created`. Fix: require `resourceType=Bundle`, `type=transaction-response`, response entries equal input entries, no error OperationOutcome, and report status buckets accurately.

- **High: repeated runs pollute HAPI.** Python emits transaction POSTs, and the harness posts them to the shared HAPI base at [147](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:147). There is no cleanup, partitioning, conditional update, or deterministic server id strategy. Fix: use an isolated HAPI instance/partition per run, purge before run, or use deterministic `PUT`/conditional operations; record DB-clean state in the report.

- **High: no environment preflight.** `run()` starts processing at [194-221](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:194) without checking HAPI, matchbox, loaded packages, or whether matchbox’s StructureMap is the local one. `path_b()` calls `$transform` by canonical URL at [127](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:127), so a stale server map can produce a false pass/fail. Fix: preflight `/metadata`, required profiles, required packages, StructureMap existence/version/hash, and abort on mismatch.

- **Medium: omitted paths are reported as 0/0 instead of skipped.** The report always renders Path A and Path B at [287-300](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:287), and CLI output always prints both at [345-347](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:345). The current `docs/E2E-REPORT.md` shows Path B `0 ok / 0 fail` for `paths=A`, which is misleading. Fix: render only selected paths, or mark unselected paths `SKIPPED`.

- **Medium: the 174-file coverage story is misleading.** `classify()` marks lowercase `metadata` examples as `legacy-variant` before checking `diagnosisOd` at [67-70](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:67). That is intentional for unsupported schema variants, but the report at [279-286](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:279) does not distinguish “oncology-like but skipped due schema variant.” Fix: report `branch`, `schema_variant`, `eligible`, and `run`, plus the `--limit` value.

- **Medium: Path A and Path B are not compared fairly.** The driver runs A and B independently at [224-249](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:224), but never compares resource counts, resource types, expected profiles, references, or mapped field parity per case. Fix: add a parity section: A-vs-B type histogram, profile coverage, missing resources, extra resources, and key field diffs.

- **Medium: there is no real overall PASS/FAIL gate.** The report lists emit/ingest/validation counts at [287-300](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:287), but never computes a verdict. A run with content errors can still look “ok” because emit and ingest passed. Fix: emit top-level `PASS`, `FAIL`, or `INCONCLUSIVE`; fail on emit/transform/structural/ingest/content errors, mark inconclusive on env/ambiguous validation failures.

- **Medium: validation results are not actionable at 174-file scale.** Aggregation keys are only `profile.split("/")[-1]` at [161](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:161), and per-case output keeps only counts at [236](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:236). Fix: store full canonical+version, file, path A/B, resourceType, id/fullUrl, profile, issue expression, and full diagnostics.

- **Medium: diagnostics are truncated too aggressively.** `first_error()` truncates at [105](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:105), and validation samples truncate at [172](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:172). The current report already cuts an error mid-sentence. Fix: keep full diagnostics in JSON and only shorten the Markdown display with a pointer to the full issue record.

- **Medium: `path_a()` has no timeout.** The Python mapper subprocess at [114-115](/Users/kohlbach/Claude/MII/genomDE2FHIR/scripts/e2e_harness.py:114) can hang and block the whole corpus. Fix: add a configurable subprocess timeout and classify it as mapper/tool timeout, not content conformance.

The biggest reputational risk is the combination of “env-only means not mapper bugs” plus missing raw env diagnostics plus `$validate` transport errors becoming `clean`. That can turn a broken validation environment into a green-looking report.
